# 백엔드 개발자 실전 기술 면접 대비 Q&A

이 문서는 프로젝트 수행 과정에서 경험한 설계, 보안, 성능, 장애 대응, 배포/운영, 테스트, 그리고 기술적 트레이드오프와 성장에 대한 깊이 있는 기술 면접 예상 질문과 STAR 기법 기반의 정밀 답변을 담고 있습니다.

## I. 설계 (Design) - 5개

### Q1. 서비스 간의 강한 결합(Strong Coupling)을 깨고 이벤트 기반의 느슨한 결합(Loose Coupling)으로 전환한 경험을 말해주세요.
* **상황(Situation):** 기존 시스템에서는 외부 AI 서버를 활용하여 리포트 PDF를 동기식으로 발급받았는데, 분석에 최소 15초 이상 지연이 걸려 전체 웹 애플리케이션 스레드가 마비되는 눈사태 장애가 잦았습니다.
* **과제(Task):** 15초 이상 걸리는 비즈니스 로직을 API 스레드 풀에서 완전히 격리하고, 메시지 유실 없이 다중 백엔드 인스턴스가 부하를 고르게 나눠 처리하도록 비동기 이벤트 아키텍처를 적용해야 했습니다.
* **행동(Action):** 분산 메시지 큐인 Kafka를 전면에 도입하여 `AiJobDispatchService`를 구현했습니다. 컨트롤러는 요청 즉시 Kafka 토픽(`ai-job-request`)으로 메시지를 발행하고 사용자에게 `202 Accepted` 응답을 뿜도록 처리했고, 백그라운드에 별도 격리된 `AiJobKafkaConsumer`가 이를 안정적으로 수신하여 생성 과정을 전담하게 했습니다.
* **결과(Result):** 동시 100건 이상의 분석 요청 폭증 시에도 서블릿 스레드 점유율은 0%를 유지했고, 응답 속도는 0.05초로 대폭 단축되어 시스템 안정성이 획기적으로 상승했습니다.
* **꼬리 질문:**
  1. **Q. Kafka 메시지 중복 소비(Idempotency) 문제가 발생한다면 어떻게 대처하시겠습니까?**
     * *A.* Redis나 RDB의 유니크 키(예: `requestId`)에 분산 락(Distributed Lock) 혹은 유니크 제약 조건을 걸어, 컨슈머가 이미 처리 완료한 요청인 경우 중복 처리를 조용히 거절(Drop)하도록 멱등성 필터를 보강하겠습니다.
  2. **Q. Kafka 브로커 전체가 다운되었을 때 대안은 무엇입니까?**
     * *A.* `dispatchReport` 메서드의 리턴값(`dispatched`)을 기준으로 안전 장치를 두어, 큐에 넣는 것이 완전히 불가능하다면 로컬 `@Async` 백그라운드 스레드 풀로 우회 처리하여 서비스 중단이 없게 막았습니다.

### Q2. 듀얼 리드(Dual-Read) 및 라이트 백(Write-Back) 패턴을 아키텍처 관점에서 어떻게 활용했나요?
* **상황(Situation):** 사용자 토큰 보안 강화를 위해 Refresh Token 해싱 방식을 기존 SHA-256에서 HMAC-SHA256으로 완전히 업그레이드해야 했습니다.
* **과제(Task):** 기존 배포를 무중단 상태로 유지하면서, 기존 로그인 세션을 쥐고 있는 사용자들이 로그아웃되거나 에러를 겪지 않게 유연하게 새로운 암호화 규격으로 갈아끼워야 했습니다.
* **행동(Action):** 데이터 조회(`loadValidToken`) 시 신규 HMAC 해시 조회를 최우선 시도하고, 실패하면 레거시 SHA-256 해시로 한 번 더 조회하는 **Dual-Read** 기법을 적용했습니다. 레거시 데이터로 조회가 정상 성공한 시점에, 자동으로 신규 HMAC 규격으로 다시 저장해주는 **Write-Back** 마이그레이션 메커니즘을 서비스 단에 내재시켰습니다.
* **결과(Result):** 무중단 배포 이후, 단 한 명의 활성 세션 드랍 없이 7일 만에 DB 및 캐시에 존재하던 모든 레거시 토큰이 안전하게 무장애 마이그레이션 완료되었습니다.
* **꼬리 질문:**
  1. **Q. RDB와 캐시(Redis) 두 저장소 간의 쓰기 일관성이 깨질 우려는 어떻게 최소화하나요?**
     * *A.* 마이그레이션 트랜잭션 도중 DB 쓰기가 실패할 경우 Spring의 `@Transactional` 롤백으로 전체 상태를 되돌리며, Redis 캐시는 DB 성공 후 저장하도록 순서를 엄격히 제어했습니다. 캐시 실패 시에는 RDB 기반의 Fallback 복구를 보장합니다.

### Q3. 도메인 주도 설계(DDD) 관점에서 엔티티 간 라이프사이클과 연관 관계를 제어한 구조를 설명해주세요.
* **상황(Situation):** 기업 보고서 생성 및 업데이트 시, 보고서(`CompanyReportsEntity`)와 각 보고서의 실제 산출물 버전(`CompanyReportVersionsEntity`) 간의 잦은 데이터 동시 쓰기로 테이블 락 및 정합성 충돌이 우려되었습니다.
* **과제(Task):** 하나의 보고서가 생성될 때마다 하위 버전 엔티티가 순차적인 버전 번호(`version_no`)를 보장하며 정합성 있게 변경을 감지하고 저장되어야 했습니다.
* **행동(Action):** `CompanyReportsEntity`를 애그리거트 루트(Aggregate Root)로 규정하고 하위 버전 발행 권한을 `CompanyReportVersionIssueService`로 철저히 캡슐화했습니다. 새 버전을 발행할 때 보고서 상위 로우에 쓰기 락을 거쳐 분기 충돌을 원천 통제했습니다.
* **결과(Result):** 대규모 분기 배치 쿼리 및 사용자 실시간 요청이 엉켜도 엔티티 정합성 깨짐 지표가 0%로 완벽하게 수렴했습니다.
* **꼬리 질문:**
  1. **Q. 락을 적극적으로 걸면 성능상 병목이 생길 텐데 이에 대한 대처는 무엇이었나요?**
     * *A.* RDB 로우에 직접 가하는 락(Pessimistic Lock) 점유 시간을 최소화하기 위해, 외부 AI 서버와의 느린 통신(15초)을 트랜잭션 밖에서 먼저 완전히 마친 뒤 파일 메타데이터 저장 및 버전 바인딩 등의 데이터 쓰기 영역만 짧은 트랜잭션으로 집중 분리했습니다.

### Q4. API 버저닝(Versioning) 전략을 프로젝트에 어떻게 구축했는지 설명하고, 그 이유를 밝혀주세요.
* **상황(Situation):** 프론트엔드와 모바일 클라이언트가 다중 버전으로 배포된 상황에서 백엔드 인증 재전송 API의 엔드포인트 규격을 대대적으로 교체해야 했습니다.
* **과제(Task):** 구버전 클라이언트의 호환성을 완전히 깨지 않고 신규 규격 API로 매끄럽게 사용자 교통 정리를 유도해야 했습니다.
* **행동(Action):** 정식 경로를 `POST /api/auth/resend-verification`로 개편하고, 기존 경로인 `GET` 요청에 대해서는 레거시 유예 경로로 유지하되 HTTP Response 헤더에 `Deprecation`, `Sunset`, `Warning` 메타데이터를 추가해 클라이언트 개발자가 콘솔 경고를 인지할 수 있게 유도했습니다.
* **결과(Result):** 구버전 경로에 대한 점진적인 교체를 안내하여, 한 달간의 유예 기간을 거쳐 트래픽 누수 없이 무사히 레거시 통신을 중단하고 구버전 코드를 최종 물리 제거했습니다.
* **꼬리 질문:**
  1. **Q. API 버저닝을 URL(Path) 방식 외에 Header 방식으로 가져가는 것과의 장단점은 무엇인가요?**
     * *A.* Header 버저닝은 URL을 오염시키지 않는 우아함이 있으나 캐싱 프록시 설정이 까다로워집니다. 반면 Path 버저닝(예: `/api/v1/...`)은 가장 직관적이고 모니터링 툴(APM)에서 엔드포인트별 트래픽을 분류 추적하기가 매우 편리해 당사는 직관적 Path 방식을 주로 선호합니다.

### Q5. 데이터 정규화와 반정규화 중 어떤 Trade-off를 고려하여 도메인 모델을 선택했나요?
* **상황(Situation):** 워치리스트와 기업 재무 정보 조회 기능에서, 복잡한 다중 조인(Join) 쿼리로 인해 대규모 사용자의 화면 렌더링에 심각한 응답 지연이 감지되었습니다.
* **과제(Task):** 3개 이상의 조인이 발생하는 무거운 화면 쿼리의 성능을 개선해야 했습니다.
* **행동(Action):** 실시간 실적 예측 정보(`PREDICTED`)를 매번 연산 조인하지 않고, 기업 보고서와 매핑되는 최신 재무 지표 캐싱용 반정규화 필드(`metric_value`) 및 Redis 직렬화 캐시 구조를 설계했습니다.
* **결과(Result):** 조인 제거 및 조회를 O(1) 수준으로 당겨옴으로써 기업 상세 대시보드 로딩 지연 시간이 1.2초에서 0.08초로 극적으로 급감했습니다.
* **꼬리 질문:**
  1. **Q. 반정규화 데이터의 정합성을 주기적으로 일치시키는 보완책은 어떻게 세우셨습니까?**
     * *A.* DB 내부 트리거 대신 스프링 애플리케이션 레벨에서 데이터가 생성/변경되는 핵심 메서드에 이벤트 리스너를 결합시켜, 마스터 데이터 업데이트 완료 즉시 반정규화 테이블 및 캐시 갱신이 자동 연쇄 반응(Write-Through/Evict)하도록 동기화했습니다.

## II. 보안 (Security) - 4개

### Q6. Refresh Token Rotation(RTR) 설계 중 토큰 재사용(Reuse Attack) 감지 시 세션을 어떻게 무효화하나요?
* **상황(Situation):** 사용자의 브라우저 세션에 보관된 Refresh Token이 해커에 의해 무단 탈취되어 악용될 수 있는 잠재적 위험에 노출되었습니다.
* **과제(Task):** 토큰 탈취에 대응하여, 해커가 구 토큰으로 새로운 세션을 훔치려 하거나 사용자가 연속적으로 재발급을 요청할 때 위협을 실시간 감지하고 완전 무효화해야 했습니다.
* **행동(Action):** 토큰 재발급(`POST /api/auth/refresh`) 시 기존 토큰의 해시값을 검증하고 즉시 사용 불가(`Revoke`) 처리함과 동시에 신규 토큰을 반환했습니다. 만약 이미 사용 불가 처리된 '구 토큰'으로 재발급 요청이 또다시 인입되면 즉각 **토큰 탈취 및 재사용 상황**으로 간주하고 해당 계정과 연결된 모든 활성 Redis 세션을 삭제(`revokeAllByUserId`)했습니다.
* **결과(Result):** 가상의 해커 시나리오 테스트 결과, 이미 교체된 구 토큰으로 재발급을 찌르는 순간 정상 사용자의 세션까지 철저히 만료 유도되어 해커의 추가 접근 권한을 완벽히 차단했습니다.
* **꼬리 질문:**
  1. **Q. 정상적인 사용자의 네트워크 일시적 지연으로 인해 재발급 요청이 중복으로 발생하여 무효화되면 억울하지 않을까요?**
     * *A.* 매우 중요한 부분입니다. 이를 위해 재발급 시점 기준 1~2초 이내의 아주 미세한 유예 시간(Grace Period)을 두어, 이 짧은 간격 내에 중복 들어온 동일 토큰 요청에 대해서는 이미 발급했던 신규 토큰 쌍을 그대로 재반환하도록 보완해 사용자 불편을 제거할 수 있습니다.

### Q7. 데이터베이스가 해킹당하더라도 토큰을 안전하게 보호할 수 있는 일방향 해시 보안 강화 전략을 설명해주세요.
* **상황(Situation):** 데이터베이스 침해 사고로 테이블 전체가 덤프 노출될 때, 평문 저장된 Refresh Token이나 이메일 인증 토큰이 그대로 노출될 경우 해커가 세션을 완전히 대규모 우회 장악할 수 있었습니다.
* **과제(Task):** 데이터베이스에 토큰 자체를 평문 저장하지 않으면서도, 실제 인입 토큰의 유효성을 정확하고 안전하게 대조/검증하는 일방향 암호화 벽을 설계해야 했습니다.
* **행동(Action):** DB에 토큰 평문 값을 즉시 폐기하고 `HMAC-SHA256` 해싱 저장 구조로 전환했습니다. 공격자가 역추적하기 어렵도록 코드나 DB에 하드코딩되지 않는 Base64 인코딩 페퍼(Pepper) 환경 변수를 구동 시점에 동적 로드하여 토큰값을 단방향 계산했습니다.
* **결과(Result):** 만약 DB 테이블 전체가 해킹당하더라도, 페퍼 키를 모르는 상태에서는 레인보우 테이블 대입으로 토큰 원본을 복구하기가 수학적으로 불가능하게 차단되었습니다.
* **꼬리 질문:**
  1. **Q. 보안용 Pepper 키를 탈취당하지 않기 위해 어떤 방식으로 관리하고 배포했습니까?**
     * *A.* Pepper 값은 절대 형상 관리 소스코드 저장소(Git)에 남겨두지 않으며 AWS Parameter Store나 환경변수(`APP_TOKEN_HASH_PEPPER_B64`) 형태로 주입하고, 형상 빌드/기동 시 형식 검증을 통해 유효하지 않으면 기동 실패(Fail-Fast)하도록 구축했습니다.

### Q8. Double Submit Cookie 패턴을 적용하여 CSRF(Cross-Site Request Forgery) 공격을 방어한 구체적 원리를 말해주세요.
* **상황(Situation):** Refresh Token과 세션 정보가 브라우저 쿠키(HttpOnly/Secure)로 자동 전송되는 구조의 특성상, 외부 악의적 피싱 사이트에서 우리 백엔드를 찌르는 CSRF 공격에 극도로 취약했습니다.
* **과제(Task):** 백엔드에서 세션을 유지하되, 악의적 도메인에서 몰래 자동 전송되는 세션 재발급/로그아웃 요청을 완벽하게 걸러내어 차단해야 했습니다.
* **행동(Action):** 더블 서브밋 쿠키 방식을 도입했습니다. 로그인 및 재발급 완료 시 클라이언트에 임의의 고유 난수인 `csrf_token` 쿠키를 새로 굽고, 프론트엔드가 HTTP API 호출 시 이 값을 읽어 `X-CSRF-Token` 요청 헤더에 같이 실어 보내도록 했습니다. 백엔드는 스프링 시큐리티 필터 단에서 이 쿠키 값과 헤더 값이 일치하는지만을 단방향 검증했습니다.
* **결과(Result):** 타 도메인 스크립트는 SOP(Same-Origin Policy) 장벽 때문에 우리 쿠키 값을 읽을 수 없으므로 헤더에 이 난수를 얹을 수 없게 되며, CSRF로 기습 들어온 불일치 요청들은 모두 403 Forbidden으로 안전하게 전량 드랍되었습니다.
* **꼬리 질문:**
  1. **Q. 서브도메인이 다를 때 쿠키 공유 및 검증에 문제가 발생하진 않나요?**
     * *A.* 서브도메인 간 공유를 위해 쿠키의 도메인(Domain) 설정을 최상위 도메인(`.domain.com`)으로 와일드카드 세팅하고, CORS(Cross-Origin Resource Sharing) 설정에 `X-CSRF-Token` 헤더를 신뢰 헤더로 명시 등록하여 안전하게 연동되도록 조치했습니다.

### Q9. 무차별 계정 대입 공격(Brute-Force Attack)을 효율적으로 감지하고 방어한 설계 구조를 설명해주세요.
* **상황(Situation):** 로그인 API를 대상으로 특정 이메일에 무작위 비밀번호를 초당 수십 회씩 난사하여 비밀번호를 캐내는 무차별 대입 공격이 유입되었습니다.
* **과제(Task):** 다중 스레드 동시 다발 공격 시 발생하는 레이스 컨디션 문제를 피하며, 특정 계정 혹은 IP의 무차별 로그인 시도를 신속하고 강력하게 격리/잠금 처리해야 했습니다.
* **행동(Action):** Redis 기반의 실패 카운팅과 계정 잠금 메커니즘인 `LoginAttemptService`를 구현했습니다. 다중 로그인 스파이 트래픽이 한꺼번에 몰릴 때 카운트 합산 오류가 나지 않도록 **Redis Lua Script**를 호출하여 실패 가산 및 실패 윈도우 시간(TTL) 갱신을 단일 원자적 명령(Atomic Operation)으로 격리 수행하도록 구축했습니다.
* **결과(Result):** 지정 횟수(예: 5회) 연속 로그인 실패 즉시 15분간 해당 사용자의 계정 로그인을 즉각 Fail-fast 차단하여, 불필요한 DB 계정 대조 쿼리 리소스 낭비도 90% 이상 절감했습니다.
* **꼬리 질문:**
  1. **Q. 해커가 특정 이메일 하나를 지속 공격해 정상 사용자가 계속 잠김 상태에 빠지게 유도하는 '도스(DoS)'성 공격은 어떻게 대응하겠습니까?**
     * *A.* 사용자 계정 이메일 외에 **접속 소스 IP 주소** 기준의 레이트 리밋(Rate Limiting) 카운터를 보완하여, 특정 사용자를 괴롭히는 해커 IP만 선제적으로 영구 차단 필터링하는 IP 차단막을 병행 구축하겠습니다.

### Q10. 자바 가상 스레드(Virtual Thread) 도입 전후로 발생한 성능 벤치마크 결과의 핵심 개선 요인은 무엇이었나요?

### Q11. 가상 스레드 운영 중 경험할 수 있는 Pinning(피닝) 이슈의 정의와 대처 방안을 말해주세요.
* **상황(Situation):** 가상 스레드를 적용하여 기쁘게 벤치마크 테스트를 돌리던 중, 암호화 처리 모듈 및 특정 레거시 코드를 관통하는 지점에서 성능 개선이 멈추고 서버가 심각하게 굳어버리는 병목 지점이 확인되었습니다.
* **과제(Task):** 가상 스레드 스케줄러를 무너뜨려 물리 하드웨어 커널 스레드까지 동반 마비시키는 피닝(Pinning) 범인을 검거하고 치유해야 했습니다.
* **행동(Action):** 가상 스레드가 코드 내부의 `synchronized` 락 영역에서 블로킹 I/O를 타는 순간 물리 캐리어 스레드에 달라붙어 떨어지지 않는(Pin) 원인을 파악했습니다. 이에 따라 병목을 일으킨 락 영역의 코드를 자바의 가상 스레드 친화적 락인 `ReentrantLock` 구조로 전면 교체 마이그레이션했습니다.
* **결과(Result):** 캐리어 스레드 고착 현상이 제거되어, 대규모 트래픽 하에서도 JVM의 모든 CPU 캐리어가 모든 가상 스레드를 부드럽게 언마운트/마운트 분산 스케줄링하게 정상 복구되었습니다.
* **꼬리 질문:**
  1. **Q. 우리가 작성한 소스 외에 3rd Party 외부 라이브러리 내부에 꽁꽁 숨겨진 피닝 현상은 어떻게 탐지할 수 있나요?**
     * *A.* JVM 구동 시 `-Djdk.tracePinnedThreads=full` 옵션을 인가하여 구동하고 벤치마크를 돌리면, 피닝 발생 즉시 락이 걸린 정확한 소스코드 라인과 스택 트레이스가 상세 콘솔 출력되므로 이를 면밀히 훑어가며 사전에 방어할 수 있습니다.

### Q14. 외부 서버 과부하 상황에서 우리 서비스 스레드 마비를 막기 위해 Circuit Breaker를 튜닝한 여정을 공유해 주세요.

* **과제(Task):** 외부 의존 장애가 인입되는 즉시 물리 통로를 단절하여 격리하고, 외부 상태가 회복되면 점진적으로 복구되는 지능형 서킷 방벽을 튜닝해야 했습니다.
* **행동(Action):** Resilience4j의 `aiServer` 인스턴스를 선언하고 `slidingWindowSize: 20`, `failureRateThreshold: 50`, `slowCallRateThreshold: 60`, 그리고 느린 요청 한계치 `slowCallDurationThreshold: 70s`를 매칭하여 최적 지점을 조율했습니다.

* **꼬리 질문:**
  1. **Q. 서킷이 OPEN에서 CLOSED로 갑자기 돌아갈 때의 과부하 충격을 방지하기 위해 어떤 상태 전이를 거치나요?**

### Q15. Bulkhead 격벽 설계를 적용하면서 `maxWaitDuration: 0ms` 옵션을 반드시 지정한 이유는 무엇인가요?
* **상황(Situation):** 동시 100명 이상의 대량 사용자가 AI 분석 리포트 생성을 한 번에 누를 때, 서킷이 터져 나가기도 전에 외부 API 처리용 세마포어(Semaphore) 대기 줄에 줄줄이 스레드들이 갇히며 다른 API까지 함께 기어가는 전조 증상이 있었습니다.
* **과제(Task):** 특정 비즈니스가 점유할 수 있는 동시 통로 개수를 명확히 20개 내외로 물리 제한하고, 이를 초과하는 오버 트래픽은 백그라운드 대기열 낭비 없이 1초의 오버헤드도 주지 않고 끊어내야 했습니다.
* **행동(Action):** Resilience4j Bulkhead 설정을 통해 동시 통로 한계를 `maxConcurrentCalls: 20`으로 제한하고, 진입 제한에 걸렸을 때 대기하는 한계 수치인 `maxWaitDuration`을 **0ms**로 극단적 차단 처리했습니다.

* **결과(Result):** 전체 서비스 레이어 테스트 구동 시간이 45초에서 **단 1.8초**로 격감하여, 코딩 직후 단축키 하나로 유닛 결함을 바로 확인하는 애자일한 극강 개발 생산성을 되찾았습니다.
* **꼬리 질문:**
  1. **Q. Mocking 위주의 테스트만 고수하면, 실제 DB 쿼리(Querydsl/JPA) 오동작에 의한 데이터 결함을 놓치지 않나요?**
     * *A.* 그렇기 때문에 테스트 영역을 이원화하여, 순수 서비스 분기는 Mock 위주의 초고속 단위 테스트로 검증하고, 복잡한 커스텀 쿼리는 `@DataJpaTest` 기반의 실제 H2 구동형 슬라이스 통합 테스트로 각개 격파하여 두 마리 토끼를 다 잡았습니다.

### Q23. 프로젝트 아키텍처 개선 과정에서 마주한 가장 큰 기술적 Trade-off는 무엇이었으며, 이를 극복한 의사결정 기준은 무엇이었나요?
* **상황(Situation):** 외부 AI 연동 속도 개선을 위해 비동기 파이프라인(Kafka)을 얹기로 결정했으나, 이로 인해 서버 인프라 유지 비용 상승, 분산 세션 상태 관리 및 네트워크 복잡도가 기하급수적으로 폭증하는 트레이드오프가 찾아왔습니다.
* **과제(Task):** "기술적 욕심"을 걷어내고, 실제 서비스 사용자 경험 개선 효과와 운영 유지 비용 간의 균형을 합리적으로 판단하여 최선의 인프라 카드를 내밀어야 했습니다.
* **행동(Action):** 동기 호출을 고수했을 때의 서블릿 고갈로 인한 **시스템 전체 다운(가동시간 붕괴)**의 리스크가 카프카 추가 운영 리소스 비용보다 훨씬 크다는 리스크 정량 비교 기준을 수립하여 카프카 적용 의사결정을 관철했습니다. 
* **결과(Result):** 비록 네트워크 복잡도는 늘었으나 서비스 연속 신뢰도를 99.99%로 격상시켰고, 개발 단계의 복잡도는 로컬 테스트 Mock 인가 토글 장치를 얹어 보완함으로써 개발자의 고통을 유연하게 상쇄했습니다.
* **꼬리 질문:**
  1. **Q. 만약 비용이 극도로 짠 스타트업이라 카프카 인프라 비용마저 허용되지 않는다면 어떤 차선책 대안을 내겠습니까?**
     * *A.* RDB나 로컬 Redis를 초경량 메시지 큐(Redis Pub/Sub)처럼 임시 사용하여 별도 미들웨어 추가 도입 없이 기존 데이터베이스 리소스 위에서 경량 비동기 큐를 내재화하는 실용적 절충안을 제안하겠습니다.

### Q24. 이번 프로젝트 전체를 관통하며 개발자로서 얻은 가장 큰 배움과, 다시 처음부터 이 설계 프로젝트를 진행한다면 개선하고 싶은 아키텍처 포인트는 무엇인가요?
* **상황(Situation):** 처음에는 단순 "기능의 빠른 구현"이 최고의 개발 경쟁력이라고 여겼으나, 프로덕션 배포 시점의 보안 구멍, 외부 장애 눈사태 폭발 등을 차례로 몸으로 겪으며 큰 좌절을 겪었습니다.
* **과제(Task):** 단순 개발자 수준을 넘어 시스템의 내결함성과 회복탄력성, 점진적 마이그레이션 안전장치를 생각하는 "프로덕션 엔지니어링 마인드"로 패러다임을 넓혀야 했습니다.
* **행동(Action):** 무리한 기술 적용보다 점진 호환(Dual-Read/Migration)을 항상 염두에 두고 코드를 짰고, Resilience4j의 격벽 차단 기준을 정밀 튜닝하며, CI/CD 스크립트까지 다이내믹 쉘 단위 테스트로 묶어 완벽한 수비형 백엔드를 만드는 체력을 길렀습니다.
* **결과(Result):** 장애 예보 단계부터 자동 복구, 무장애 마이그레이션까지 전체 생명주기를 완벽 제어하는 시야의 백엔드 개발자로 다시 태어났습니다. 다시 돌아간다면 AI 리포트 변환 포맷을 PDF 바이너리 직접 통신 대신 메타데이터 JSON 포맷 규격화로 파이프라인을 교체하여 대역폭 효율을 20배 이상 늘리는 아키텍처로 바꾸고 싶습니다.
* **꼬리 질문:**
  1. **Q. PDF 통신 대신 JSON 메타데이터 방식을 적용할 때 클라이언트(Front-end) 측 부담은 어떻게 조절할 것인가요?**
     * *A.* 백엔드에서 raw 바이너리를 내려주는 것 대신 분석 JSON을 인가하고 프론트엔드 단에서 PDF 생성 라이브러리(PDFKit 등)를 활용하게 렌더링 부하를 클라이언트로 분산시켜, 백엔드 CPU 리소스 부하를 극단적으로 청소할 수 있습니다.

### Q25. 3인 이상 다수 개발자가 공동 작업하는 모놀리식 백엔드 코드베이스에서, 브랜치 전략 및 코드 퀄리티를 유지한 본인의 리더십이나 노하우를 말씀해 주세요.
* **상황(Situation):** 공동 소스코드 병합(Merge) 및 기능 단위 테스트 부재로 인해, 한 명의 개발자가 마이너 수정을 반영해 푸시하면 다른 팀원의 샌드박스가 깨지는 빌드 단절 현상이 잦았습니다.
* **과제(Task):** 소스코드가 공용 master에 통합되는 모든 게이트 단계에서 개발 규칙 불일치 및 빌드 버그를 선제적이고 기계적으로 차단할 룰을 안착해야 했습니다.
* **행동(Action):** PR(Pull Request) 시 무조건 CI 빌드 및 전체 JUnit 단위 테스트를 성공해야만 `Merge` 버튼이 활성화되도록 깃허브 브랜치 보호 정책(Branch Protection Rules)을 강제 적용했고, PR 검토 시 도메인 애그리거트 경계를 깨는 코드 디자인이 감지되면 단호하게 개선 리뷰를 건네는 수호 규칙을 리드했습니다.
* **결과(Result):** 협업 한 달 차 이후부터 메인 브랜치의 빌드 통과율이 100% 철통 보장되었고, 병합 충돌 해결에 소모되던 오버헤드 시간이 80% 이상 전량 개선되어 모두가 신뢰하는 소스 생태계를 달성했습니다.
* **꼬리 질문:**
  1. **Q. 코드 리뷰 과정에서 팀원 간 의견 충돌(예: 가독성 vs 최적화 성능)이 격하게 일어날 때 어떤 기준으로 갈등을 완화했습니까?**
     * *A.* 주관적 직관보다 객체지향 설계 원칙(SOLID)의 기준을 공유하며, 무엇보다 **성능 테스트 벤치마크 데이터(TPS/MS 지표)**를 계측해서 정량적 수치로 가시적 답을 도출하는 팩트 중심의 엔지니어링 토론으로 갈등을 스마트하게 풀어갔습니다.
